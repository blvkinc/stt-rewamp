 Set The Table MVP Technical Requirements Document - Working Draft v0.5 Prepared from Uploaded MVP notes, clarification Q&amp;A, and prior STT source pack Prepared on 2026-03-22 Document intent Normalize confirmed MVP scope into a buildable venue-first website + admin platform, including clarified recurring-event inventory plus package-level and whole-event availability behavior. Scope guard: This draft treats the uploaded MVP file, the later venue-first clarifications, the STT-Questions responses, and the recurring-event inventory/availability clarifications as the controlling launch scope. 1. Purpose and framing This document translates the uploaded MVP notes and the later clarification pass into a structured technical requirements draft that can guide product design, data modeling, API planning, engineering sequencing, and future merchant-portal expansion. The intent is to preserve confirmed launch scope only, avoid unconfirmed scope expansion, and capture the technical implications required to make the requested MVP operable in production. The launch operating model in this draft is a customer-facing website plus an STT admin portal. The MVP is venue-first, merchant self-service is deferred, KYB/business-license collection is deferred, and venue records created by STT admins must remain compatible with a future merchant-claim layer. 2. MVP outcome The MVP should deliver a working public customer booking experience backed by a Set The Table admin portal that performs all venue-side setup and operational functions for launch. Customers can create accounts, sign in via email or social auth, browse and filter listings, favorite venues/events, book across one or more events, pay securely where payment is required, and access bookings, receipts, and cancellation requests. STT admins can create and manage venues, venue payout details, events, open-ended recurring event series, event occurrences, packages, package combinations, event-level pricing and availability rules, default package inventory, occurrence-level overrides, booking data, customer data, and reporting. The platform structure must remain intentionally scalable so later phases can introduce merchant organization accounts above venues, merchant login, controlled multi-venue access, venue claims, and KYB/business-license collection without redesigning the MVP core. 3. MVP design principles Principle Application to this MVP Admin-first launch All venue-side creation and management flows are executed by STT admins at launch. Venue-first MVP model Venue is the operational unit in MVP. Merchant organization accounts are deferred until the future merchant portal. Future merchant compatibility The data model must allow a future merchant organization layer to attach above existing venues without forcing a launch dependency on merchant records. Normalized package architecture Canonical package identity remains separate from event-package configuration so the same package can behave differently per event and occurrence. Occurrence-level inventory Recurring events should use a default event-package inventory and availability model that each occurrence inherits unless an explicit occurrence override is created, while still allowing event-level occurrence closure when the whole event date is unavailable. Default + exception occurrence control Open-ended recurring events should not require a finite end date or manual stock setup for every future date; default event-package behavior remains the source of truth and only exceptions are stored at occurrence level. Operational availability controls STT admins must be able to mark a package sold out or unavailable for a specific occurrence/date and optionally stop selling it after a specified time without changing the base package definition. Per-event-package payment control Each package attached to an event can independently use full payment, deposit, or no upfront payment. Mixed-cart privacy by venue A single customer order may contain multiple venue bookings, but venue-facing data, notifications, and exports must expose only that venue's relevant booking lines. Structured customer data ownership Reusable customer identity/contact/residence data must be separated from booking-specific inputs such as hair and makeup and additional notes. Venue-level financial ownership Any bank account or payout configuration belongs to the venue, not to a future merchant organization layer. Event-level availability control Whole event occurrences must also support sold-out and unavailable states so a specific date can be closed at event level without manually closing every package. 4. Actor model Actor Launch access Primary responsibilities in MVP Notes Customer Active Website signup/login, discovery, favorites, cart, booking, payment, receipts, cancellation requests Website only for MVP; no native app or in-app notifications. STT Super Admin Active Full system control, account administration, venue/event/package control, approvals, reporting, destructive actions Highest privilege level. STT Admin Active Operational management of venues, events, packages, rules, inventory, bookings, and reporting Broad operational access below Super Admin. STT Editor Active Content and catalog setup such as venues, events, media, FAQs, and non-destructive updates Editorial/admin support role. STT Bookings &amp; Customer Manager Active Booking handling, customer information access, cancellation handling, operational support Should be constrained to bookings/customer surfaces. Merchant organization (future phase) Deferred Registers, claims one or more venues, manages user access across venues, and submits KYB/business-license information Not created in MVP. Merchant is a future top-level organization account above venues. 5. Functional requirements 6.1 Customer requirements ID Requirement CUS-01 The platform shall allow a customer to create a Standard account via email signup/login. CUS-02 The platform shall support Google login and Facebook login for customers. CUS-03 The platform shall support email verification, forgot password, and password reset flows. CUS-04 The platform shall provide a website-based customer dashboard/profile area with access to bookings, past receipts, favorites, and cancellation requests. CUS-05 The platform shall display searchable and filterable venue and event listings using the confirmed launch filters: Day, Day or Night, Cuisine, Budget, Area, Brand / Hotel, Rating, Type, Offers, Birthday Offers, Family Friendly / Animal Friendly, and Indoors or Outdoor. CUS-06 The platform shall display event detail pages containing the venue as organizer, category/tag information, location with embedded map, event date/time, accessibility information, description, ordered media, reusable FAQs, and SEO/public metadata. CUS-07 The platform shall allow customers to add packages from multiple events and multiple venues into a single cart before checkout. CUS-08 The platform shall display a bookable calendar with available dates and times at occurrence level. CUS-09 The platform shall apply promo codes and valid pricing rules/discounts during checkout. CUS-10 The platform shall support one order containing booking lines with different payment modes, charging only the payable amount through Stripe while still creating valid booking records for lines that require no upfront payment. CUS-11 The platform shall generate booking confirmations and e-tickets after successful booking finalization and make them accessible from customer history. CUS-12 The platform shall send the required customer/system emails: signup verification, welcome email, booking confirmation, receipt / e-ticket, payment failed, reminder before event, and cancellation / refund update. CUS-13 The platform shall preserve venue-level privacy when a single order contains bookings across multiple venues; venue-facing views, notifications, and exports shall expose only the booking data relevant to that venue. CUS-14 The platform shall capture reusable customer identity/contact/residence data including first name, last name, day of birth, month of birth, year of birth, gender, nationality, type of stay, country of residence, city/emirate, area, address line 1, email address, WhatsApp area code, WhatsApp number, and phone number. CUS-15 The platform shall support structured customer classification fields for Teacher and Cabin Crew rather than leaving them as loose free-text only. CUS-16 The platform shall store booking-specific inputs such as hair and makeup and additional notes separately from the reusable customer account/profile data. CUS-17 The platform shall not require guest-detail capture per booking line in MVP. 6.2 Admin and operating requirements ID Requirement ADM-01 The platform shall provide an STT admin portal and dashboard. ADM-02 The platform shall support the MVP admin roles of Super Admin, Admin, Editor, and Bookings &amp; Customer Manager. ADM-03 The platform shall not require merchant account creation, merchant login, venue claims, KYB, or business-license collection for MVP operations. ADM-04 The admin portal shall support venue creation and management as the operational onboarding unit for MVP. ADM-05 The admin portal shall support venue-level bank / payout details. ADM-06 The platform shall support one active global commission configuration in MVP, where the commission mode can be flat or percentage but only one mode is active at a time. ADM-07 Commission shall be calculated per booking line / package and payouts shall remain manual in MVP. ADM-08 The admin portal shall support event creation, editing, recurrence management, and lifecycle management under a venue. ADM-09 The platform shall support the internal workflow states Draft, Pending, and Approved for admin-managed records. ADM-10 The admin portal shall support reusable venue-level package creation and package-combination setup. ADM-11 The admin portal shall support event-level configuration of rules, pricing, availability, payment mode, and inventory for a package once that package is attached to an event. ADM-12 The admin portal shall support access to booking data, event data, and customer data for operational use. ADM-13 The admin portal shall support venue-scoped booking notifications, exports, and operational views when one customer order spans multiple venues. ADM-14 The admin portal shall support customer analysis by booking, event, and venue context. ADM-15 The admin portal shall support reusable venue-level FAQ management so events can attach existing FAQs without repetitive manual entry. ADM-16 The platform shall support the required email integrations for launch system events, including admin internal notifications. 6.3 Event, package, rule, and inventory requirements ID Requirement CAT-01 Event categories shall be managed from a global table controlled only by STT admins. CAT-02 Each event shall have exactly one primary category and may have multiple subcategories. EVT-01 Events shall support one-off, daily, weekly, monthly, and yearly occurrence patterns. EVT-02 After selecting an occurrence pattern, the system shall allow the applicable date, day, and time structure to be set and managed. EVT-03 The platform shall support single-occurrence edits, single-occurrence cancellation, and single-occurrence price/rule overrides, with guardrails to avoid unsafe changes to past occurrences or already-booked future occurrences. EVT-04 The platform shall support blackout dates / skipped dates for recurring events. EVT-05 Venue information shall be linked to the event and displayed on the event details page through the venue relationship; for MVP the event organizer is the venue. EVT-06 Event images and videos shall support ordered sequencing. EVT-07 Event FAQs shall be attachable from a venue-managed reusable FAQ set. EVT-08 Events and venues shall support SEO/content fields including meta title, meta description, custom slug, and Open Graph / social share image. EVT-09 Recurring event series may remain open-ended without a mandatory end date. EVT-10 STT admins shall be able to mark a specific event occurrence/date as sold out or unavailable at the whole-event level. EVT-11 STT admins shall be able to set a specific event occurrence/date as not available or closed for sale without manually disabling each package. EVT-12 Whole-event occurrence availability shall take precedence over package-level availability so that if an occurrence is closed or sold out, no package under that occurrence remains bookable. PKG-01 Base packages shall support the required launch fields: name, slug, package type, default guest count, and base price. PKG-02 Package type shall support Individual, Couple, and Group, where Individual defaults to guest count 1, Couple defaults to guest count 2, and Group requires an entered guest count. PKG-03 Packages shall be venue-specific and visible only within the scope of the owning venue. PKG-04 The base packages table shall retain canonical package definitions needed for cross-event reporting within a venue, while event-package configuration remains separate. PKG-05 The same base package shall be able to use different prices, rules, availability, payment modes, and inventory states when attached to different events or different occurrences of an event. PKG-06 Package combinations / bundled packages are part of MVP. PKG-07 Each package attached to an event shall have its own payment mode configuration: full payment, deposit, or no upfront payment. PKG-08 Where deposit mode is used, the event-package configuration shall store both the deposit amount and the full amount reference required for booking and reporting. RULE-01 The MVP rule engine shall support Ladies, Gents, Kids, age-based pricing, early-bird pricing with cutoff date, time-based availability, day-based availability, availability constraints, and inventory constraints. RULE-02 Multiple rules may apply to the same booking so long as they are not conflicting; the platform shall enforce conflict constraints rather than assuming only one rule can apply. INV-01 Inventory shall be tracked per package per event occurrence. INV-02 For recurring events, default inventory and baseline availability shall be configured on the event-package and inherited by occurrences unless an explicit occurrence-level override exists. INV-03 STT admins shall be able to override package inventory for a specific event occurrence/date without changing the default inventory for the overall recurring series. INV-04 STT admins shall be able to mark a package as sold out or unavailable for a specific event occurrence/date even where default inventory still exists. INV-05 STT admins shall be able to close a package for a specific date or stop selling it after a specified time for an occurrence. INV-06 For recurring events, one occurrence being sold out or unavailable shall not imply that another occurrence is sold out or unavailable. INV-07 Cart items shall temporarily reserve stock for a default hold period of 1 hour, and that hold duration shall be configurable by STT admins. INV-08 The MVP shall not depend on an automated shared-capacity engine across packages to manage exception cases; operational package availability controls must be sufficient for launch. BOOK-01 A customer may purchase multiple quantities, multiple packages from the same event, and packages from different events or venues within one order. BOOK-02 Guest-detail capture per booking line is not required for MVP. 6.4 Reporting and analytics requirements ID Requirement AN-01 The platform shall provide revenue reports by day, month, and year. AN-02 The platform shall provide booked-events reports by day, month, and year. AN-03 The platform shall provide customer analytics views. AN-04 The platform shall provide venue analytics views. AN-05 The admin portal shall allow customer information to be inspected in relation to booking, event, and venue activity. 7. Non-functional requirements for MVP viability The MVP architecture shall be extensible enough to add the merchant portal later without redesigning the core data model. Any future merchant organization layer should attach above venues rather than replacing venue-level ownership. The platform shall preserve referential integrity across venue, event, occurrence, package, inventory, booking, and payment data. Inventory updates shall be transaction-safe to reduce oversell risk during checkout. Stripe webhook handling shall be treated as a source of truth for final payment state. Administrative actions that affect listings, prices, inventory, or bookings should be auditable. Recurring event support shall use a structured recurrence/occurrence model with default event-package inventory and availability inheriting across occurrences unless a specific occurrence override is created; it shall also support whole-event occurrence closure without relying on manually duplicating entire event records for every date. Operational stock and availability management shall support manual per-occurrence sold-out and unavailable states at both package and whole-event level, including date-specific closure and time-based package cutoffs, without requiring an automated shared-capacity engine in MVP. The reporting layer shall use normalized data rather than UI-derived calculations. Email integrations shall be event-driven and support failure logging/retry handling. Venue-facing operational outputs shall enforce data isolation so a venue cannot see booking lines belonging to another venue within the same customer order. Mixed-payment orders shall remain internally consistent, allowing one order to contain full-payment, deposit, and no-upfront lines while preserving accurate payment-state tracking per booking line. Commission calculations shall be derived per booking line from the single active global commission configuration, and manual payout workflows shall use venue-level payout ownership. 10.
